## Purpose

End-to-end testing of the MetaMergeDelete → Rollback round-trip on representative test data revealed five issues ranging from a fundamental `file_system.relative` computation bug to cleanup gaps in MetaMergeDelete's post-processing phase. The test data covers edge cases including Unicode filenames (`💀💀💀.png`), special characters (`[a] {b} (c).pdf`), yt-dlp sidecars, empty nested directories, a 728 MB binary archive, duplicate files, and pre-existing `_meta.json` / `_meta2.json` artifacts from prior indexing runs.

**Summary of issues:**

1. **`file_system.relative` includes the target directory name** (§5.6 violation). The indexer computes relative paths from the *parent* of the target directory rather than the target itself, causing all relative values to carry an unwanted leading path component. This cascades into rollback creating an extra nesting level in the restored tree.
2. **Stale `_meta2.json` from prior runs causes duplicate/misplaced restores.** Rollback picks up orphaned `_meta2.json` files that survived MetaMergeDelete (because they were excluded from traversal at Layer 1), producing a second restore of the same content file at an incorrect path.
3. **Prior-run `_meta.json` files survive MetaMergeDelete.** The same Layer 1 exclusion mechanism that enables Issue 2 also leaves behind `_meta.json` artifacts. These don't directly corrupt rollback but clutter the tree and violate user expectations.
4. **Rollback does not validate session provenance of loaded metadata.** When multiple `_meta2.json` files reference the same content file with conflicting `file_system.relative` values, rollback trusts all of them blindly.
5. **Empty directory trees are not reconstructed by rollback.** This is expected behavior but needs to be documented as an intentional design boundary.

### Implementation Ordering

Items in this document have dependencies that constrain execution order:

1. **Section 1** (`file_system.relative` fix) — Foundational. Changes the value of every relative path in every `_meta2.json` produced by the indexer. Must be implemented first because all downstream consumers (`_write_inplace_tree`, `_rename_tree`, rollback) depend on the relative path format. This is a **breaking change** for existing `_meta2.json` files.
2. **Section 2** (MetaMergeDelete stale-file cleanup) — Eliminates the source of Issues 2 and 3. Depends on Section 1 only in the sense that testing should use the corrected relative paths, but the cleanup logic itself is independent.
3. **Section 3** (Rollback session-ID hardening) — Defensive layer that protects rollback from stale metadata even when Section 2's cleanup is not available (e.g., when operating on `_meta2.json` files produced by older indexer versions). Can be implemented in parallel with Section 2.
4. **Section 4** (Spec and documentation updates) — Must be executed last, after all code changes are verified.

## 1. Fix `file_system.relative` Path Computation (Priority: Critical)

### 1.1. Problem Statement

The `file_system.relative` field for every entry produced by a directory indexing operation includes the target directory's own name as a leading path component. The spec (§5.6) explicitly states:

> *For the root item itself (the target of the indexing invocation), this is `"."`.*

When indexing `A:\Code\shruggie-indexer-testing\data`, the root directory entry should have `relative = "."` and a child at `data\images\slippers.gif` should have `relative = "images/slippers.gif"`. Instead, the root gets `relative = "data"` and the child gets `relative = "data/images/slippers.gif"`.

### 1.2. Evidence

The rollback log demonstrates the cascading effect. The first directory created is the spurious nesting level:

```
Directory created: ...\data2\data
```

Every restored file lands under this nested `data\` prefix:

```
Restored: slippers.gif → ...\data2\data\images\slippers.gif
Restored: 123.nfo      → ...\data2\data\123.nfo
```

Expected (after fix):

```
Restored: slippers.gif → ...\data2\images\slippers.gif
Restored: 123.nfo      → ...\data2\123.nfo
```

### 1.3. Root Cause

In `src/shruggie_indexer/core/entry.py`, both `build_directory_entry()` and `build_file_entry()` default the index root to the parent of the target:

```python
index_root = _index_root if _index_root is not None else path.parent
```

When `index_path()` dispatches to `build_directory_entry()` for a directory target, it does not pass `_index_root`. The default resolves to `path.parent` — the parent of the directory being indexed. All `relative_forward_slash(child_path, index_root)` calls then compute paths relative to the parent, which includes the target directory's name as a prefix.

`_write_inplace_tree()` and `_rename_tree()` in `src/shruggie_indexer/cli/main.py` work around this by reconstructing paths via `root_path.parent / entry.file_system.relative` — compensating for the extra prefix by anchoring to the parent directory. This workaround masks the bug during normal indexing operations, but rollback's `target_dir / entry.file_system.relative` does not apply the same compensation, causing the extra nesting level.

### 1.4. Required Changes

**A. Fix `index_root` default in `build_directory_entry()`.**

Change the default from `path.parent` to `path`:

```python
# BEFORE (broken):
index_root = _index_root if _index_root is not None else path.parent

# AFTER (correct):
index_root = _index_root if _index_root is not None else path
```

This makes the root directory entry's relative `"."` (per the spec) and all children's relatives are anchored to the target directory itself.

**Do NOT change `build_file_entry()`'s default.** When `build_file_entry()` is called standalone (single-file index via `index_path()`), `path.parent` is the correct root — the file's relative should be its filename alone (e.g., `"slippers.gif"`). When `build_file_entry()` is called from within `build_directory_entry()`, the corrected `index_root` is passed via the `_index_root` parameter, so the default is not used.

**B. Update `_write_inplace_tree()` path reconstruction.**

```python
# BEFORE (compensating for parent-based relative):
item_path = root_path.parent / entry.file_system.relative

# AFTER (relative is now anchored to root_path directly):
item_path = root_path / entry.file_system.relative
```

Apply this change to both the file branch and the directory branch of `_write_inplace_tree()`.

**C. Update `_rename_tree()` path reconstruction.**

```python
# BEFORE:
item_path = root_path.parent / entry.file_system.relative

# AFTER:
item_path = root_path / entry.file_system.relative
```

**D. Update the docstring/comment in `_rename_tree()`.**

Remove or rewrite the comment that states: *"Path reconstruction uses `root_path.parent / relative` ... because `file_system.relative` is computed relative to the parent of the target directory (the `index_root`)."* Replace with a note that relative is computed from the target directory itself.

**E. Update existing unit tests.**

Tests in `tests/unit/test_serializer.py` (and any other test files) that construct `IndexEntry` objects with `file_system.relative` values including the root directory name must be updated. For example, a test entry with `relative="root/sub/file.txt"` should become `relative="sub/file.txt"` (or `relative="."` for the root entry).

Search all test files for the pattern `file_system=FileSystemObject(relative=` and verify each value conforms to the corrected convention.

### 1.5. Breaking Change Notice

This change alters the `file_system.relative` value in every `_meta2.json` and `_directorymeta2.json` file produced by the indexer. Existing sidecar files produced by prior versions will contain the old (parent-based) relative paths. Downstream consumers must be aware:

- **Rollback** is the primary consumer. Section 3 of this document adds defensive handling for the old format.
- **`_write_inplace_tree()` and `_rename_tree()`** are updated in this section.
- **No other known first-party consumers** exist yet. `shruggie-vault`, `shruggie-catalog`, and `shruggie-sync` are unimplemented.

## 2. MetaMergeDelete: Clean Up Stale Metadata Files from Prior Runs (Priority: High)

### 2.1. Problem Statement

MetaMergeDelete's post-processing (Stage 6) only deletes files that were consumed as sidecars during the current run. Files matching `metadata_exclude_patterns` (Layer 1) — specifically `_meta.json`, `_meta2.json`, `_directorymeta.json`, and `_directorymeta2.json` files from prior indexer runs — are excluded from traversal before sidecar discovery can see them. They are never queued for deletion and accumulate silently across repeated runs.

This creates two concrete problems:

1. **Stale `_meta2.json` files corrupt rollback** (observed in testing). The source directory contained `FeedsExport_meta2.json` (old naming convention, without `.7z` extension) alongside the current run's `yA94EC04C88CC4D8273F15BE034C18FEE.7z_meta2.json`. Rollback loaded both, restoring `FeedsExport.7z` twice at conflicting paths — once at the target root (728 MB, wrong location) and once nested correctly.
2. **Orphaned `_meta.json` files clutter the tree.** After MetaMergeDelete, the user expects a clean directory containing only content files (renamed to `storage_name`) and their current-run `_meta2.json` sidecars. Instead, six stale `_meta.json` files remained in the post-MMD tree.

### 2.2. Evidence

Post-MetaMergeDelete input tree retained the following stale files:

| Stale file | Location | Convention |
|-----------|----------|------------|
| `123.nfo_meta.json` | `not-empty/` | `_meta.json` (v1 output) |
| `123.txt_meta.json` | `not-empty/` | `_meta.json` (v1 output) |
| `[a] {b} (c)_1.pdf_meta.json` | `not-empty/pdf/pdf/` | `_meta.json` (v1 output) |
| `[a] {b} (c)_2.pdf_meta.json` | `not-empty/pdf/pdf/` | `_meta.json` (v1 output) |
| `FeedsExport_meta.json` | `not-empty/` | `_meta.json` (v1, no `.7z` extension) |
| `FeedsExport.7z_meta.json` | `not-empty/` | `_meta.json` (v1, with `.7z` extension) |
| `FeedsExport_meta2.json` | `not-empty/` | `_meta2.json` (prior v2 run, no `.7z` extension) |

The `FeedsExport_meta2.json` file is the direct cause of the duplicate rollback restore. Its `file_system.relative` value (from a prior run with different path conventions or a different target) did not match the current run's value, so rollback treated it as a distinct entry for the same content file.

### 2.3. Root Cause

The two-layer exclusion mechanism works as designed for *preventing re-indexing* of prior output — but it has no *cleanup* component. The operational flow:

1. `list_children()` applies `metadata_exclude_patterns` (Layer 1). Files matching `_(meta2?|directorymeta2?)\.json$` are removed from the file list.
2. Sidecar discovery operates on the `siblings` list, which is the *full* file list from `list_children()`. Since stale metadata files were already removed at Layer 1, they are invisible to sidecar discovery.
3. MetaMergeDelete queues only files that were successfully parsed as sidecars during the current run.
4. Stage 6 deletes only queued files.
5. Stale metadata files are never queued, never deleted.

### 2.4. Required Changes

**Add a Stage 7 cleanup phase to the MetaMergeDelete pipeline.** After Stage 6 (sidecar deletion) completes, perform a targeted scan for stale metadata artifacts:

**A. Scan each traversed directory for files matching `metadata_exclude_patterns`.**

After the current run's `_meta2.json` sidecars are written (Stage 5) and consumed sidecars are deleted (Stage 6), iterate over each directory that was traversed during indexing. For each directory, enumerate files matching the `metadata_exclude_patterns` regex. Any file that matches and is NOT one of the current run's output files is a stale artifact.

**B. Identify current-run output files.**

The current run's output files can be identified by:
- Filename pattern: `{storage_name}_meta2.json` or `{directory_name}_directorymeta2.json`
- Or more robustly: maintain a set of `Path` objects representing every sidecar file written during Stage 5. Pass this set to the Stage 7 cleanup routine.

**C. Delete stale files with logging.**

For each stale file found, log at `INFO` level and delete:

```
INFO  Stale metadata artifact removed: not-empty/FeedsExport_meta2.json
```

If deletion fails (permissions, etc.), log at `WARNING` and continue — do not abort the operation.

**D. Implementation location.**

The cleanup logic should be implemented in `src/shruggie_indexer/core/entry.py` (alongside the existing Stage 6 post-processing) or in a new helper function called from the orchestration path in `cli/main.py` and `gui/app.py`. The cleanup must run *after* the current run's sidecar writes and *after* the Stage 6 consumed-sidecar deletion, but *before* the operation reports completion.

**E. Guard with `meta_merge_delete` flag.**

This cleanup only runs when `meta_merge_delete=True`. Plain MetaMerge and plain Index operations do not delete anything.

| `src/shruggie_indexer/cli/main.py` | Add Stage 7 cleanup call after `_write_inplace_tree()` and Stage 6 deletion. |
| `src/shruggie_indexer/gui/app.py` | Add Stage 7 cleanup call in the GUI background job. |

- Rollback of the post-cleanup directory produces no duplicate or misplaced restores.

## 3. Rollback: Session-ID Validation and Legacy Compatibility (Priority: High)

### 3.1. Problem Statement

`discover_meta2_files()` (or the rollback loader) collects all `*_meta2.json` files in a directory without validating whether they belong to the same indexing session. When stale metadata from prior runs coexists with current-run output — as observed in the FeedsExport.7z duplicate restore — rollback silently loads conflicting entries for the same content file.

Even with Section 2's cleanup in place, rollback must be resilient against this scenario because:

- Users may run rollback against directories where cleanup was not performed (e.g., `_meta2.json` files from an older indexer version that predates the cleanup feature).
- Users may manually place `_meta2.json` files from different sessions into the same directory.
- Third-party tools may produce `_meta2.json` files with varying conventions.

### 3.2. Required Changes

**A. Session-ID consistency check in `plan_rollback()`.**

After `load_meta2()` assembles the flat list of entries, and before planning begins, check for content-hash collisions — entries where two or more `IndexEntry` objects share the same content hash (`hashes.md5` or `hashes.sha256`) but have different `file_system.relative` values. When a collision is detected:

1. If both entries have `session_id` values, prefer the entry whose `session_id` is the most common across all loaded entries (the "majority session"). Discard the minority entry with a `WARNING` log:

```
WARNING: Duplicate content hash {hash} found in multiple sessions.
  Keeping: {relative} (session {session_id})
  Discarding: {relative} (session {session_id})
```

2. If only one entry has a `session_id`, prefer that entry over the one without.

3. If neither has a `session_id`, or both have the same `session_id` but different `relative` values, keep the first encountered and log a `WARNING`.

**B. Legacy `file_system.relative` prefix detection.**

To support `_meta2.json` files produced before the Section 1 fix, add a detection heuristic to `plan_rollback()`:

1. After loading entries, inspect the root entry (the one with `type = "directory"` or the first entry if all are files). If the root entry's `file_system.relative` is a single path component (not `"."`) — e.g., `"data"` — this indicates the legacy prefix convention.
2. Strip the detected prefix from all entries' `file_system.relative` values before planning. Replace the root entry's relative with `"."`. Log at `INFO`:

```
INFO: Detected legacy relative path prefix 'data'. Stripping prefix for rollback.
```

This ensures rollback produces correct output regardless of whether the `_meta2.json` was produced before or after the Section 1 fix.

| File | Nature of change |

| `tests/unit/test_rollback.py` | Add tests for collision detection, session-ID tiebreaking, and legacy prefix stripping. |

### 3.5. Acceptance Criteria

- When loading `_meta2.json` files with the legacy prefix convention (root relative is a directory name, not `"."`), rollback strips the prefix and restores files directly under `target_dir/` without extra nesting.

### 4.3. Empty Directory Omission: Intentional Design Boundary

Add a note to the spec (in §6.11 Rollback Operations, or in a "Known Limitations" subsection) and to the documentation site (`docs/user-guide/rollback.md`) explicitly stating:

> **Empty directories are not reconstructed by rollback.** The indexer catalogs *files* and their containing directory hierarchy. Directories that contain no files (directly or transitively) produce no entries in `_meta2.json` output and therefore cannot be restored by rollback. This is intentional — `shruggie-indexer` focuses on the preservation and cataloging of files, not the replication of directory trees. File tree replication (including empty directory preservation) is out of scope.

This should appear in both the spec and the user-facing rollback documentation so that users understand the boundary and don't report it as a bug.
