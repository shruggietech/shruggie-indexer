## Purpose and Context

The v0.2.0 release introduced the v3 schema version bump, including new `_meta3.json` sidecar naming, the `encoding` field, and associated rollback enhancements. Post-release manual testing revealed two critical regressions where v2-era hardcoded references survived the v3 sprint:

1. **Rollback discovery and loading are v2-only.** `discover_meta2_files()` globs exclusively for `*_meta2.json`, and `load_meta2()` rejects any `schema_version` other than `2`. This renders the rollback engine completely non-functional against v3 output.
2. **In-place sidecar rename uses v2 naming.** `rename_inplace_sidecar()` constructs sidecar paths using `_meta2.json` suffixes, so the rename phase renames a file that does not exist (the v3 sidecar is named `_meta3.json`) and leaves the actual sidecar orphaned with its original-name base.

Both bugs are the same class of failure: the v3 sprint updated the "write" side of sidecar handling (traversal exclusion, serializer output, stale cleanup) but missed the "read" and "rename" consumers that locate sidecars by filename pattern. The spec (§5.13, §6.11, §18.2.3) was updated to claim v3 compatibility in the rollback engine, but the code was not.

Additionally, the existing test suite did not catch either regression before release. There is no integration test that performs a full index-then-rollback round-trip, which is the most basic end-to-end verification of the pipeline's reversibility contract. This sprint adds that test coverage as a mandatory gate.

Finally, the CI release pipeline has had recurring build failures in cloud environments. This sprint adds a pre-release smoke test to the local workflow and documents the PyInstaller hidden-import verification procedure.

**Governing spec sections:** §5.12 (sidecar naming), §5.13 (backward compatibility), §6.9 (output routing), §6.10 (rename coordination), §6.11 (rollback operations), §13.5.1 (release pipeline), §14.1-14.3 (testing strategy).

## Section 1: Rollback Sidecar Discovery — v3 Support

**Depends on:** Nothing.

### Problem Statement

`discover_meta2_files()` in `core/rollback.py` uses a hardcoded glob pattern `*_meta2.json` to find sidecar files. After the v3 sprint, all new sidecar output uses `_meta3.json` naming. The function finds zero sidecars in any directory that has been processed by v0.2.0.

### Evidence

Testing log, rollback invocation:

```
Rollback plan: entries=0, target_dir=...\03-rollback, source_dir=...\02-mmd
Rollback complete: 0 restored, 0 duplicates, 0 sidecars, 0 skipped, 0 failed
```

The 02-mmd tree contains 11 `_meta3.json` files and zero `_meta2.json` files (all v2 sidecars were cleaned as stale artifacts during the indexing run).

### Root Cause

The v3 sprint updated the exclusion patterns and the serializer output naming but did not update the rollback discovery function. The function name itself (`discover_meta2_files`) reflects the v2-era assumption.

### Required Changes

**File:** `src/shruggie_indexer/core/rollback.py`

1. Rename `discover_meta2_files()` to `discover_sidecar_files()`. Retain `discover_meta2_files` as a deprecated alias that calls the new function, so that any downstream consumers (vault module, user scripts) do not break.
2. Change the glob pattern from `*_meta2.json` to match both v2 and v3 sidecars. The recommended approach is two glob passes (`*_meta2.json` + `*_meta3.json`), merged and sorted. Do NOT use a regex-based scan; `Path.glob()` is sufficient and consistent with the existing implementation style.
3. Update the docstring to reflect multi-version discovery.

**File:** `src/shruggie_indexer/core/__init__.py`

4. Add `discover_sidecar_files` to the exports. Keep `discover_meta2_files` in exports for backward compatibility.

**File:** `src/shruggie_indexer/__init__.py`

5. Add `discover_sidecar_files` to the public API exports and `__all__`. Keep `discover_meta2_files` in `__all__` for backward compatibility.

### Acceptance Criteria

- `discover_sidecar_files(directory)` returns paths for both `_meta2.json` and `_meta3.json` files.
- `discover_meta2_files(directory)` continues to work as an alias.
- Given a directory containing only `_meta3.json` sidecars, `discover_sidecar_files()` returns a non-empty sorted list.
- Given a directory containing a mix of v2 and v3 sidecars, both are returned.
- Recursive mode (`recursive=True`) discovers v3 sidecars in subdirectories.

## Section 2: Rollback Loader — v3 Schema Acceptance

**Depends on:** Section 1 (discovery must find the files before the loader can parse them).

### Problem Statement

`load_meta2()` in `core/rollback.py` contains a hard gate:

```python
version = data.get("schema_version")
if version != 2:
    msg = (
        f"Unsupported schema version in {path}: expected 2, got {version}. "
        "v1 sidecar files must be migrated to v2 format before rollback."
    )
    raise IndexerConfigError(msg)
```

This rejects all v3 sidecar files, even though v3 is a strict superset of v2 and the rollback engine's data model (`IndexEntry`, `MetadataEntry`) already supports all v3 fields.

### Root Cause

Same as Section 1. The schema version gate was not updated during the v3 sprint.

### Required Changes

**File:** `src/shruggie_indexer/core/rollback.py`

1. Change the version gate to accept both 2 and 3. Use a set-based check: `if version not in {2, 3}`. Keep the rejection of v1 (`schema_version` absent or `1`) and any future unrecognized versions.
2. Update the error message to reflect the accepted versions: `"expected 2 or 3, got {version}"`.
3. Rename `load_meta2()` to `load_sidecar()`. Retain `load_meta2` as a deprecated alias. Update the docstring references from `_meta2.json` to the version-agnostic term "sidecar file."
4. Update all internal callers of `load_meta2()` within `core/rollback.py` (the directory-mode recursive call) to use the new name.

**File:** `src/shruggie_indexer/core/__init__.py`

5. Add `load_sidecar` to exports. Keep `load_meta2`.

**File:** `src/shruggie_indexer/__init__.py`

6. Add `load_sidecar` to exports and `__all__`. Keep `load_meta2`.

**File:** `src/shruggie_indexer/cli/rollback.py`

7. Update the CLI rollback command to call `load_sidecar()` instead of `load_meta2()`. If the CLI currently imports `load_meta2` by name, update the import.

**File:** `src/shruggie_indexer/gui/app.py`

8. Same update for the GUI rollback path.

### Acceptance Criteria

- `load_sidecar(path_to_v3_sidecar)` succeeds and returns a list of `IndexEntry` objects.
- `load_sidecar(path_to_v2_sidecar)` continues to work.
- `load_sidecar(path_to_v1_sidecar)` raises `IndexerConfigError` with a clear message.
- `load_meta2()` continues to work as an alias.
- `load_sidecar(directory_with_v3_sidecars)` discovers and loads all v3 sidecars.
- Mixed v2+v3 directories: both are loaded and returned in a combined list.

## Section 3: In-Place Sidecar Rename — v3 Naming

**Depends on:** Nothing (can be executed in parallel with Sections 1-2).

### Problem Statement

`rename_inplace_sidecar()` in `core/rename.py` constructs sidecar paths using hardcoded `_meta2.json` suffixes:

```python
old_sidecar = original_path.parent / f"{original_path.name}_meta2.json"
new_sidecar = original_path.parent / f"{storage_name}_meta2.json"
```

Since v0.2.0, the serializer writes `_meta3.json` files. The rename function looks for a `_meta2.json` file that does not exist, returns `None`, and the actual `_meta3.json` sidecar is left with its original-name base, orphaned from its hash-renamed content file.

### Evidence

Testing log sequence for `123.nfo`:

```
Inplace sidecar written to ...\02-mmd\123.nfo_meta3.json
File renamed: 123.nfo → y65830E2E332651F6235D52A8553E1864.nfo
Inplace sidecar renamed: 123.nfo_meta2.json → y65830E2E332651F6235D52A8553E1864.nfo_meta2.json
```

The v3 sidecar is written. The content file is renamed. The rename function renames a stale v2 sidecar (which is then cleaned as stale), while the v3 sidecar (`123.nfo_meta3.json`) stays untouched with the wrong base name. The 02-mmd tree confirms this pattern across all files.

### Root Cause

`rename_inplace_sidecar()` was written during the v2 era and hardcodes the `_meta2.json` suffix. The v3 sprint did not update this function.

### Required Changes

**File:** `src/shruggie_indexer/core/rename.py`

1. Change the sidecar suffix from `_meta2.json` to `_meta3.json` in both the `old_sidecar` and `new_sidecar` path constructions.
2. Update the function's docstring to reflect v3 naming.

**File:** `src/shruggie_indexer/cli/main.py`

3. Update any inline comments that reference `_meta2.json` in the rename/inplace pipeline to say `_meta3.json`. Grep for `meta2` in the comments of `_rename_tree()` and the Stage 6 block. The code references should already be calling `rename_inplace_sidecar()` without hardcoding the suffix, but the comments are stale.

**File:** `src/shruggie_indexer/gui/app.py`

4. Same comment update for the GUI's rename pipeline comments.

### Acceptance Criteria

- After indexing with `--rename --inplace`, every hash-renamed content file (`y*.ext`) has a corresponding `y*.ext_meta3.json` sidecar with the matching base name.
- No orphaned original-name `_meta3.json` sidecars remain.
- The `DEBUG`-level log message reads `Inplace sidecar renamed: {original}_meta3.json → {storage_name}_meta3.json`.

## Section 4: Index-Then-Rollback Integration Test

**Depends on:** Sections 1, 2, 3 (the round-trip test exercises the fixed code paths).

### Problem Statement

The test suite has no integration test that exercises the full pipeline round-trip: index a directory (with rename, inplace, MetaMergeDelete), then roll back from the produced sidecars, then verify the restored output matches the original. This is the most basic end-to-end verification of the reversibility contract, and its absence allowed two critical regressions to ship in v0.2.0.

### Required Changes

**New file:** `tests/integration/test_roundtrip.py`

This file MUST contain, at minimum, the following test cases. All tests use `tmp_path` for isolation.

#### Test 1: `test_index_rename_rollback_roundtrip`

1. Create a fixture directory in `tmp_path` containing:
   - A text file with known content.
   - A subdirectory containing two image files (binary, identical content to exercise dedup).
   - A JSON sidecar companion file for one of the images (to exercise MetaMergeDelete).
2. Copy the fixture to a working directory (the "mmd" directory).
3. Run `index_path()` with `rename=True`, `output_inplace=True`, `meta_merge_delete=True`.
4. Assert: content files have been hash-renamed (`y*` prefix).
5. Assert: `_meta3.json` sidecar files exist with the correct hash-renamed base name (not the original name).
6. Assert: no `_meta2.json` files exist.
7. Assert: duplicates have been removed.
8. Run `discover_sidecar_files()` on the mmd directory (recursive). Assert non-empty result.
9. Run `load_sidecar()` on the mmd directory (recursive). Assert `len(entries) > 0`.
10. Run `plan_rollback()` with the loaded entries, targeting a new empty directory.
11. Assert: `plan.stats.files_to_restore > 0`.
12. Run `execute_rollback()`.
13. Assert: `result.restored > 0` and `result.failed == 0`.
14. Assert: the restored directory contains files with the original names.
15. Assert: restored text file content matches the original byte-for-byte.

#### Test 2: `test_v3_sidecar_discovery`

1. Create a directory containing a mix of `_meta2.json` and `_meta3.json` files (manually written, minimal valid JSON with the correct `schema_version`).
2. Call `discover_sidecar_files()`. Assert both v2 and v3 files are returned.
3. Call `load_sidecar()` on each. Assert no exceptions for valid v2 and v3 files.
4. Create a file with `schema_version: 1`. Call `load_sidecar()`. Assert `IndexerConfigError`.

#### Test 3: `test_inplace_sidecar_rename_v3`

1. Create a file, run `build_file_entry()` to produce an `IndexEntry`.
2. Write an inplace `_meta3.json` sidecar using `write_inplace()`.
3. Call `rename_item()` on the file.
4. Call `rename_inplace_sidecar()` on the original path.
5. Assert: the `_meta3.json` sidecar has been renamed to `{storage_name}_meta3.json`.
6. Assert: no `_meta2.json` files exist in the directory.

**Pytest markers:** `@pytest.mark.integration`, `@pytest.mark.rollback`

Register `rollback` as a new marker in `pyproject.toml` `[tool.pytest.ini_options]` markers list.

### Acceptance Criteria

- All three tests pass on a clean run.
- `test_index_rename_rollback_roundtrip` exercises the full pipeline without mocking any core module.
- The tests would have failed against the v0.2.0 codebase (i.e., they are genuine regression detectors, not tautological).
