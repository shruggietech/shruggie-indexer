## Decisions and Rationale

### 1. v3 Schema Bump (Breaking Change)

**Decision:** Increment `schema_version` from 2 to 3 rather than adding `encoding` as an additive v2 extension.

**Rationale:** The `encoding` field introduces a new top-level object on `IndexEntry` and a new field on `MetadataEntry`. Combined with `timestamps.created_source` and `json_indent`, this constitutes sufficient structural change to warrant a version bump. The sidecar naming convention (`_meta3.json`) provides immediate visual differentiation between v2 and v3 output.

### 2. Encoding Detection as Default-On

**Decision:** File encoding detection is enabled by default, with opt-out via `--no-detect-encoding` and `--no-detect-charset`.

**Rationale:** The primary consumer (Metadexer vault) requires encoding metadata for hash-perfect byte reversal. Making it default-on ensures all new indexing operations produce reversal-ready output without requiring users to discover an opt-in flag. The `chardet` dependency is lightweight and the performance cost is negligible (first 8 KB read per file).

### 3. Separate Charset vs. Full-Encoding Opt-Out

**Decision:** Two distinct flags rather than a single disable toggle.

**Rationale:** BOM and line-ending detection are deterministic and zero-dependency. `chardet`-based charset detection is probabilistic and adds a dependency. Users who want the deterministic metadata but distrust heuristic detection can disable only charset inference while retaining BOM/line-ending data.

### 4. Sidecar Naming Convention Change

**Decision:** Output filenames change from `_meta2.json`/`_directorymeta2.json` to `_meta3.json`/`_directorymeta3.json`.

**Rationale:** Prevents ambiguity when v2 and v3 sidecars coexist in the same directory. Traversal exclusion patterns updated to match all three generations (v1, v2, v3).

### 5. Symlink Type Enum Deferral

**Decision:** The `"symlink"` type enum value candidate was explicitly deferred to a future v4.

**Rationale:** Adding a third type value would alter the semantic meaning of an existing field and require reworking symlink handling throughout traversal, entry construction, and serialization. The risk was disproportionate to the encoding-focused objective of this sprint.

### 6. v0.2.0 Semver Selection

**Decision:** Minor version bump (0.1.x → 0.2.0) rather than patch.

**Rationale:** The v3 schema is a breaking change for consumers that parse `schema_version`. Semver requires at minimum a minor bump for breaking changes in pre-1.0 software.

### 7. Test Skip for v2 Schema Conformance

**Decision:** Added skip decorators to v2 schema conformance tests for entries that now emit `schema_version=3`.

**Rationale:** v2 conformance tests validate against the v2 JSON Schema which mandates `"schema_version": 2`. Entries produced by the current codebase emit `3`, which correctly fails v2 validation. Rather than deleting the v2 tests (which remain valuable for validating older sidecar files), skipping them for v3 entries preserves test coverage for both schema generations.

## Known Issues Carried Forward

Two known issues were documented in the v0.2.0 changelog and remain open in the `[Unreleased]` section:

1. **GUI: MetaMergeDelete output field reports false `_directorymeta3.json` output** — The GUI completion summary incorrectly displays the directory metadata path when directory metadata writing is disabled. Actual file operations are correct; only the display string is affected.

2. **Core: Windows `.lnk` shortcut sidecar misassociation** — Windows `.lnk` directory shortcut files are incorrectly paired as sidecars of unrelated sibling files. Binary content is correctly indexed and rollback works, but the association logic needs investigation. This was confirmed during post-release manual testing (see `.archive/20260321-001-Testing.log`, line 9: `yt-dlp - Shortcut.lnk` incorrectly associated with `123.nfo`).
