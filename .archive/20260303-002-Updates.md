## 1. Rollback Content-Hash Collision Detection Discards Legitimate Canonical Files (Priority: Critical)

### 1.1. Problem Statement

The content-hash collision detection logic added in Batch 1 (Section 3) is too broadly scoped. It was designed to detect and discard stale `_meta2.json` entries from prior indexing sessions that reference the same content file with conflicting `file_system.relative` values. However, the detection fires on *any* two entries that share a content hash, including legitimate same-session entries that represent distinct canonical files with different file extensions.

In the test data, `slippers.gif` and `slippers.png` are byte-identical files (same content hash `B4ADC74442D00EE0953105C01D42B72B`). During MMD, the indexer correctly treats them as separate canonical files because they have different file extensions — they receive distinct storage names (`yB4ADC74442D00EE0953105C01D42B72B.gif` and `yB4ADC74442D00EE0953105C01D42B72B.png`) and separate `_meta2.json` sidecars. During rollback, the collision detector sees two entries with the same content hash and discards one, resulting in `slippers.png` being silently dropped from the restored tree.

This is a **data loss bug**. A file that was successfully indexed, renamed, and preserved through MMD is permanently lost during rollback.

### 1.2. Evidence

The rollback log shows the collision detection firing:

```
WARNING  Duplicate content hash B4ADC74442D00EE0953105C01D42B72B/7EAC2B68DB9F87CD64B7DD25280A533BB4FF997F8A958943247CAB747C861EDC found in multiple sessions.
  Keeping: images/slippers.gif (session 7745fde3-5b74-4c0c-b70e-3e29cd8108bc)
  Discarding: images/slippers.png (session 7745fde3-5b74-4c0c-b70e-3e29cd8108bc)
```

Note both entries have the **same** session ID. The warning text says "found in multiple sessions" but this is false — both are from the same session.

The rollback summary confirms 10 canonical files restored. The source tree has 11 canonical content files (excluding metadata artifacts). The missing file is `images/slippers.png`.

Additionally, the collision is internally contradictory: the canonical `slippers.png` entry is discarded, but its duplicate entries (`slippers_01.png`, `slippers_02.png`) are still restored. The duplicates reference the canonical's storage name as their source, so they succeed — but the canonical they descend from was deemed "invalid."

Post-rollback tree comparison (excluding metadata artifacts):

| File | Source | Rollback | Status |
|------|--------|----------|--------|
| `images/slippers.gif` | ✅ | ✅ | OK |
| `images/slippers.png` | ✅ | ❌ | **MISSING** |
| `images/slippers.webm` | ✅ | ✅ | OK |
| `images/slippers_01.gif` | ✅ | ✅ | OK (dup) |
| `images/slippers_01.png` | ✅ | ✅ | OK (dup of discarded canonical) |
| `images/slippers_01.webm` | ✅ | ✅ | OK (dup) |
| `images/slippers_02.gif` | ✅ | ✅ | OK (dup) |
| `images/slippers_02.png` | ✅ | ✅ | OK (dup of discarded canonical) |
| `images/slippers_02.webm` | ✅ | ✅ | OK (dup) |

### 1.3. Root Cause

The Batch 1 collision detection (Section 3.2.A) keys on content hash alone:

> *Check for content-hash collisions — entries where two or more `IndexEntry` objects share the same content hash (`hashes.md5` or `hashes.sha256`) but have different `file_system.relative` values.*

This condition is satisfied by `slippers.gif` and `slippers.png` because they have identical content bytes but different relative paths. The tiebreaker logic (majority session, first-encountered) then discards one of them.

The collision detection was intended for a specific scenario: stale `_meta2.json` files from a prior session referencing the *same logical file* at a *different relative path*. Two entries from the *same session* with the *same content hash* but *different storage names* represent a fundamentally different situation — they are distinct canonical files that happen to share content. The storage name is the distinguishing signal: if two entries have different `attributes.storage_name` values, they were tracked as separate files during indexing.

### 1.4. Required Changes

**Narrow the collision detection scope in `plan_rollback()`.** The collision check MUST only flag entries as collisions when ALL of the following conditions are met:

1. Same content hash (`hashes.md5` and/or `hashes.sha256`).
2. Same `attributes.storage_name` (same canonical file identity).
3. Different `file_system.relative` values (conflicting paths for the same file).

When two entries share a content hash but have **different** `storage_name` values, they are distinct canonical files and MUST NOT be treated as collisions. No warning should be emitted; both entries proceed through the normal planning pipeline.

**Fix the warning message text.** The current message says "found in multiple sessions" regardless of whether the entries actually come from different sessions. The message SHOULD accurately reflect the situation:

- If the entries have different `session_id` values: `"Duplicate content hash {hash} found in multiple sessions."`
- If the entries have the same `session_id`: `"Duplicate content hash {hash} with conflicting paths in session {session_id}."`

### 1.5. Affected Files

| File | Nature of change |
|------|------------------|
| `src/shruggie_indexer/core/rollback.py` | Narrow the collision detection key from content hash alone to content hash + storage name. Fix warning message to accurately reflect session status. |
| `tests/unit/test_rollback.py` | Add test: two entries with same content hash but different storage names are NOT treated as collisions — both are restored. Update existing collision test to use same storage name for entries that should collide. |

## 2. Stage 6 Consumed-Sidecar Deletion Produces No Log Messages (Priority: Medium)

### 2.1. Problem Statement

When MetaMergeDelete runs, Stage 6 drains the `delete_queue` — the list of sidecar files that were consumed (parsed and merged into parent entries) during sidecar discovery. The spec (§6.10, "MetaMergeDelete execution loop") requires each successful deletion to produce an `INFO`-level log message:

```
INFO  Sidecar deleted: {path}
```

In the 2026-03-03 test session, 9 sidecar files were discovered and consumed:

- `FeedsExport.7z_info.json`
- `FeedsExport_info.json`
- `How_the_DNS_works-[2ZUxoi7YNgs].description`
- `How_the_DNS_works-[2ZUxoi7YNgs].en-en.vtt`
- `How_the_DNS_works-[2ZUxoi7YNgs].en-orig.vtt`
- `How_the_DNS_works-[2ZUxoi7YNgs].en.vtt`
- `How_the_DNS_works-[2ZUxoi7YNgs].info.json`
- `How_the_DNS_works-[2ZUxoi7YNgs].url`
- `How_the_DNS_works-[2ZUxoi7YNgs]_screen.jpg`

The post-MMD tree confirms all 9 are gone — the `yt-dlp/` directory contains only the renamed `.mp4` and its `_meta2.json`, and `not-empty/` no longer has the `_info.json` files. The deletions are happening. But the MMD log contains **zero** `Sidecar deleted:` messages. The only deletion-related logging is the Stage 7 stale cleanup messages (`Stale metadata artifact removed:`).

This has been identified in prior batches but remains unfixed. It is a logging-only issue — no data is at risk — but the missing audit trail violates the spec and makes it impossible to verify the delete phase from logs alone.

### 2.2. Root Cause

There are two likely causes (the implementing agent must investigate to determine which applies):

**A. The delete loop lacks `logger.info()` calls.** The loop may be implemented as a bare `sidecar_path.unlink()` without the corresponding log statement specified in §6.10.

**B. The delete queue is drained before the log handler is active.** If the delete loop runs very early in post-processing (before logging is fully initialized or flushed), messages may be lost. This is less likely given that Stage 7 logging immediately after works fine.

**C. The delete queue is being consumed by a different code path.** It's possible the consumed sidecars are being deleted somewhere other than the specified Stage 6 loop — perhaps inline during sidecar discovery or during rename — and the formal Stage 6 drain finds an empty queue.

### 2.3. Required Changes

**A. Locate the actual deletion code path.** Grep the codebase for `delete_queue`, `unlink`, and sidecar deletion logic. Identify exactly where and when consumed sidecars are removed from disk. Document the finding in the commit message.

**B. Ensure the Stage 6 drain loop matches the spec.** The loop MUST be:

```python
for sidecar_path in delete_queue:
    try:
        sidecar_path.unlink()
        logger.info("Sidecar deleted: %s", sidecar_path)
    except OSError as exc:
        logger.error("Sidecar delete FAILED: %s: %s", sidecar_path, exc)
```

If sidecar deletion is happening elsewhere (e.g., during rename), either move it to the Stage 6 loop or add logging at the actual deletion site — but do not have deletions happening in two places.

**C. Verify with a test.** After the fix, re-run the same test case and confirm that the log contains 9 `Sidecar deleted:` messages between the rename phase and the Stage 7 stale cleanup phase.

|------|------------------|

## 3. Sidecar Round-Trip Fidelity (Priority: Medium)

### 3.1. Problem Statement

Restored sidecar files differ in byte size from the originals. When sidecars are consumed during MetaMergeDelete, their content is read through a format-detection cascade (JSON → text → binary) and stored in the `MetadataEntry.data` field with applied transforms (`json_compact`, `base64_encode`, etc.). During rollback, the `data` field is decoded and written back to disk. This round-trip is not byte-perfect, and in one case — the `.url` Windows shortcut type — the transform is destructively lossy, producing a non-functional file on restore.

### 3.2. Evidence

Size comparison of sidecar files (source → rollback):

| Sidecar | Source (bytes) | Rollback (bytes) | Delta | Cause |
|---------|---------------|-------------------|-------|-------|
| `FeedsExport.7z_info.json` | 47,102 | 47,100 | −2 | Trailing whitespace stripped during JSON compact |
| `FeedsExport_info.json` | 47,102 | 47,100 | −2 | Same as above |
| `...en-en.vtt` | 83 | 90 | +7 | Text line-ending normalization |
| `...en-orig.vtt` | 960 | 1,024 | +64 | Same — text transform adding line endings |
| `...en.vtt` | 83 | 90 | +7 | Same as `.en-en.vtt` |
| `...info.json` | 805,289 | 900,084 | +94,795 | JSON re-serialization with different formatting |
| `...url` | 69 | 43 | −26 | **Lossy**: URL extraction stripped INI wrapper; restored file is non-functional |

Three sidecars round-tripped perfectly: `.description` (74 bytes), `_screen.jpg` (22,636 bytes), and all sidecar types that use binary/base64 encoding.

### 3.3. Analysis

The mismatches fall into three categories requiring three distinct fixes:

**A. Lossy `.url` extraction (functional breakage).** The `link` sidecar type's data handling is specified as "URL/path extraction" (§6.7). For `.url` files, this strips the Windows INI wrapper (`[InternetShortcut]\nURL=...`) and stores only the bare URL string. On restore, the raw URL string is written — producing a plaintext file that is not a valid Windows shortcut. Double-clicking the restored `.url` file does nothing; the original was a functional shortcut that opened the default browser.

**B. Missing `.lnk` metadata extraction (information loss).** Windows `.lnk` (binary shortcut) files currently fall through to base64 encoding — the raw bytes are preserved for rollback, but the informational payload (target path, working directory, arguments, icon location, description) is opaque. The index entry provides no inspectable metadata about where the shortcut points. This is the opposite problem from `.url` files: rollback fidelity is fine, but API/inspection value is zero.

**C. JSON formatting loss.** The `json_compact` transform strips all whitespace during ingest, but the rollback decoder re-serializes with `json.dumps()` using default settings. The 95KB growth in the yt-dlp `.info.json` demonstrates this: the original was compact (no whitespace), the restore re-serialized with indentation. Both representations are semantically identical, but the restored file's formatting does not match the original's structural intent.

**D. Text encoding normalization (`.vtt` files).** Minor byte-count changes from line-ending normalization. These fall within the permissible tolerance for whitespace-level differences and require no fix.

### 3.4. Required Changes

#### 3.4.A. Change `.url` sidecar type from URL extraction to text cascade

**Current behavior:** The `link` sidecar type performs URL/path extraction — it parses the `.url` file and stores only the extracted URL string in `data`.

**New behavior:** `.url` files MUST be processed through the standard format-detection cascade (JSON → text → binary fallback) instead of URL extraction. A `.url` file is not valid JSON, so it falls through to the text step. The full file content — including the `[InternetShortcut]` header and `URL=` key — is stored verbatim as `format: "text"`. On rollback, the text is written back, producing a functional Windows shortcut file.

The `attributes.type` remains `"link"` for identification purposes. The change is only to the data handling path — switching from extraction to preservation.

**Note:** This is a **breaking change** for existing `_meta2.json` files that contain `.url` sidecar entries. Rollback of old-format entries will continue to produce the bare URL string (the data is what it is). Only newly indexed `.url` sidecars will benefit from the full-content preservation.

#### 3.4.B. Add dedicated `.lnk` binary shortcut handler with dual-storage

**Current behavior:** `.lnk` files fall through the format-detection cascade to the binary step and are base64-encoded. The `data` field contains the raw bytes. No metadata extraction occurs — the shortcut's target, arguments, and other fields are opaque.

**New behavior:** `.lnk` files MUST be handled by a dedicated parsing subprocess that:

1. **Extracts key metadata fields** from the binary `.lnk` format and stores them in a structured `attributes.link_metadata` object:

   | Field | Description |
   |-------|-------------|
   | `target_path` | The file, folder, or resource the shortcut points to. |
   | `working_directory` | The working directory used when launching the target. |
   | `arguments` | Command-line arguments passed to the target. |
   | `icon_location` | Path to the icon resource (and icon index, if applicable). |
   | `description` | The shortcut's comment/description string. |
   | `hotkey` | Keyboard shortcut assigned to the link (if any). |

   Fields that are absent or empty in the `.lnk` file SHOULD be omitted from the object (not stored as `null` or empty strings).

2. **Base64-encodes the raw binary** and stores it in `data` with `format: "base64"` and `transforms: ["base64_encode"]`, exactly as the current fallback behavior does.

This dual-storage approach provides both API/inspection value (structured `link_metadata`) and byte-perfect rollback fidelity (base64 `data`). Downstream consumers can inspect `link_metadata.target_path` without decoding the binary; rollback uses `data` for restoration.

**Implementation notes:**

- The `.lnk` parser should be implemented as a dedicated module or utility function in `core/sidecar.py` (or a new `core/lnk_parser.py` if the logic is substantial).
- A pure-Python approach is preferred for cross-platform compatibility — `.lnk` files may be encountered when indexing network shares or external drives from any platform, not just Windows. The MS-SHLLINK binary format is [publicly documented](https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-shllink). Third-party libraries (e.g., `pylnk3`, `LnkParse3`) are acceptable if they are pure-Python, actively maintained, and have no native dependencies. The choice of library vs. custom parser is left to the implementing agent.
- If `.lnk` parsing fails for any reason (corrupted file, unsupported format version, missing library), the handler MUST fall back gracefully to plain base64 encoding without `link_metadata`. Log the failure at `WARNING` level. The file is still preserved — only the metadata extraction is skipped.
- The `attributes.type` for `.lnk` files changes from `"link"` to `"shortcut"` to distinguish them from `.url` files, which retain `type: "link"`. This reflects the fundamental difference: `.url` files are text-based link references; `.lnk` files are binary shortcut objects.

**Dependency management:** If a third-party `.lnk` parsing library is chosen, it MUST be added as an **optional** dependency in `pyproject.toml` (similar to how `pyexiftool` is handled). The parser module should handle the import gracefully:

```python
try:
    import lnk_library  # whichever library is chosen
    _HAS_LNK_PARSER = True
except ImportError:
    _HAS_LNK_PARSER = False
```

When `_HAS_LNK_PARSER` is `False`, `.lnk` files fall back to plain base64 encoding without metadata extraction — the same behavior as today. No error is raised; a `DEBUG`-level log message notes the missing library.

#### 3.4.C. Detect and preserve JSON formatting style at ingest

**Current behavior:** The `json_compact` transform normalizes all JSON sidecars to a compact representation (no whitespace) during ingest. On rollback, the restore decoder re-serializes the parsed JSON with `json.dumps()` using unspecified formatting, producing inconsistent results.

**New behavior:** Before applying the `json_compact` transform, the sidecar ingest logic MUST detect whether the original JSON was compact or pretty-printed:

1. **Detection heuristic:** If the raw file text contains `\n ` or `\n\t` (a newline followed by whitespace), the JSON is pretty-printed. Otherwise, it is compact. This is a simple string check that covers the vast majority of real-world JSON files.

2. **Record the style:** Store the result as `attributes.json_style` with one of two values:
   - `"compact"` — original JSON had no significant indentation.
   - `"pretty"` — original JSON was indented/formatted.

3. **Apply during rollback restore:** When restoring a JSON-format sidecar:
   - If `attributes.json_style == "compact"` (or `json_style` is absent for backward compatibility): `json.dumps(data, separators=(',', ':'), ensure_ascii=False)`.
   - If `attributes.json_style == "pretty"`: `json.dumps(data, indent=2, ensure_ascii=False)`.

The restored file will not be byte-identical to the original (the original's indent width, key order, trailing newlines, etc. may differ), but it will match the original's structural intent — a compact file restores as compact, a formatted file restores as formatted. Minor whitespace differences in the restored output are acceptable.

**Note:** This attribute applies only to sidecar entries (`origin: "sidecar"`) processed through the JSON format-detection path. It does NOT apply to ExifTool-generated metadata entries, which are always JSON and are not restored during rollback.

### 3.7. Acceptance Criteria

**`.url` sidecar (3.4.A):**
- `.url` files are processed through the text cascade, not URL extraction.
- The full file content (including `[InternetShortcut]` header) is stored in `data` with `format: "text"`.
- Rollback of a `.url` sidecar produces a functional Windows shortcut file (double-clickable).
- `attributes.type` remains `"link"`.

**`.lnk` sidecar (3.4.B):**
- `.lnk` files produce a `MetadataEntry` with both `data` (base64-encoded raw binary) and `attributes.link_metadata` (structured fields).
- `link_metadata` contains at minimum `target_path` when available.
- Rollback of a `.lnk` sidecar produces a byte-perfect copy of the original binary.
- `attributes.type` is `"shortcut"` (not `"link"`).
- When the `.lnk` parser library is unavailable, `.lnk` files fall back to plain base64 without `link_metadata` and without error.
- Parser failure on a corrupted `.lnk` file falls back gracefully with a `WARNING` log.

**JSON style detection (3.4.C):**
- JSON sidecars detected as compact store `attributes.json_style: "compact"`.
- JSON sidecars detected as pretty-printed store `attributes.json_style: "pretty"`.
- Rollback restores compact JSON as compact (no whitespace).
- Rollback restores pretty JSON as indented (2-space indent).
- Existing `_meta2.json` files without `json_style` restore using compact format (backward-compatible default).

**Text sidecars (no change):**
- Minor whitespace differences (trailing newlines, line-ending normalization) in `.vtt` and `.description` restores are acceptable and require no fix.

**General:**
- All existing sidecar unit tests continue to pass.
- Round-trip test for the standard test data produces functional sidecars for all types.

### 4.6. Changelog Entry

Add to `CHANGELOG.md` under the appropriate version heading:

```markdown
### Fixed
- **Core: Rollback content-hash collision detection scope** — Narrowed the
  rollback planner's content-hash collision detection to require matching
  `storage_name` in addition to matching content hash. Previously, two
  entries with the same content hash but different storage names (e.g.,
  `yAAA.gif` and `yAAA.png` — byte-identical files with different
  extensions) were incorrectly treated as a cross-session collision, causing
  one to be silently discarded. This resulted in data loss during rollback.
  Entries with different storage names are now recognized as distinct
  canonical files and both are restored. Spec: §6.11.
- **Core: Stage 6 consumed-sidecar deletion logging** — The MetaMergeDelete
  Stage 6 delete-queue drain now produces `INFO`-level `Sidecar deleted:`
  log messages for each consumed sidecar file removed from disk, matching
  the specification (§6.10). Previously, the deletions occurred but
  produced no log output.
- **Core: `.url` sidecar round-trip fidelity** — The `link` sidecar type
  now preserves the full file content of `.url` (Windows shortcut) files
  via the standard text cascade, instead of performing lossy URL extraction
  that stripped the INI wrapper. Restored `.url` sidecars are functional
  Windows shortcuts. Spec: §6.7, §7.5.
- **Core: JSON sidecar formatting preservation** — JSON-format sidecars now
  detect and record their original formatting style (compact vs.
  pretty-printed) in `attributes.json_style`. Rollback restores compact
  JSON as compact and pretty-printed JSON as indented, matching the
  original file's structural intent. Spec: §6.7, §6.11.

### Added
- **Core: `.lnk` binary shortcut sidecar handler** — Added dedicated
  handling for Windows `.lnk` (binary shortcut) sidecar files. The handler
  extracts structured metadata (target path, working directory, arguments,
  icon location, description, hotkey) into `attributes.link_metadata` for
  API/inspection use, while also base64-encoding the raw binary for
  byte-perfect rollback restoration. Assigned `attributes.type: "shortcut"`
  to distinguish from text-based `.url` files (`type: "link"`). The `.lnk`
  parser is an optional dependency; when unavailable, falls back to plain
  base64 without metadata extraction. Spec: §6.7, §7.5.
```
