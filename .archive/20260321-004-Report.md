### Phase 5 — PyInstaller Hidden-Import Audit and CI Hardening (Section 5)

**New file:** `hooks/hook-chardet.py` (+18 lines)

**Root cause:** chardet v7.2.0 is compiled with mypyc, which creates a top-level `.pyd` runtime support module in `site-packages` with a hash-based name (`0deeb2fec52624e647be__mypyc.cp312-win_amd64.pyd`). This module is invisible to PyInstaller's standard collection mechanisms (`collect_all`, `collect_submodules`, `collect_data_files`, `collect_dynamic_libs`). At runtime the bundled executable raised `ModuleNotFoundError: No module named '0deeb2fec52624e647be__mypyc'`, which chardet caught internally and silently degraded to "chardet is not installed".

**Fix:** Created a custom PyInstaller hook that:
1. Calls `collect_all('chardet')` for standard package data and submodules.
2. Glob-searches for `*__mypyc*` files relative to chardet's parent directory in `site-packages`.
3. Adds discovered mypyc runtime modules as both binaries and hiddenimports.

**Modified files:**
- `shruggie-indexer-cli.spec`: Added `hookspath=["hooks"]` and `"chardet"` to hiddenimports.
- `shruggie-indexer-gui.spec`: Same changes.

## Decisions and Rationale

### 1. Function Rename with Deprecated Alias Retention

**Decision:** Renamed `discover_meta2_files()` → `discover_sidecar_files()` and `load_meta2()` → `load_sidecar()`, retaining the old names as deprecated forwarding aliases in both the implementation and `__all__`.

**Rationale:** The old names are v2-specific and misleading now that the functions handle both v2 and v3 sidecars. However, downstream consumers (including `metadexer`) may be using the old names. Retaining aliases avoids a breaking change in a patch release. Aliases remain in `__all__` so that `from shruggie_indexer import *` continues to work. Deprecation will be enforced at 1.0.0.

### 2. Set-Based Version Gate (`{2, 3}`) Instead of Range Check

**Decision:** Used `if version not in {2, 3}` rather than `if version < 2 or version > 3` or a minimum-version check.

**Rationale:** An explicit set ensures that future schema versions (v4, v5) require a deliberate code change to accept. A range or minimum check (`version >= 2`) would silently accept unknown future versions that might have incompatible structural changes. The set approach is fail-safe: new versions require an implementation update before they can be loaded.

### 3. Custom PyInstaller Hook for mypyc Runtime Discovery

**Decision:** Created a dedicated `hooks/hook-chardet.py` with glob-based mypyc module discovery rather than hardcoding the hash-named module.

**Rationale:** The mypyc runtime module has a hash-based name (`0deeb2fec52624e647be__mypyc`) that is not stable across chardet versions or build environments. A glob pattern (`*__mypyc*`) ensures the hook works regardless of the specific hash value. This is more resilient than hardcoding the module name, which would break on the next chardet release.

### 4. Single Release Commit

**Decision:** All sprint work (sections 1–7) was committed as a single release commit rather than per-section commits.

**Rationale:** The sprint plan specified a single `release: prepare v0.2.1` commit message. All seven sections form a cohesive patch — the bug fixes, the tests that validate them, the documentation that describes them, and the version bump are logically inseparable. Atomic commits also simplify `git bisect` if a future regression needs to be tracked.

### 5. Known Issues Moved from Unreleased to v0.2.1

**Decision:** The two open known issues (GUI false output, `.lnk` misassociation) were moved from the `[Unreleased] > To-Do` section to `[0.2.1] > Known Issues`.

**Rationale:** The sprint plan explicitly required this move. These issues were documented during v0.2.0 but not addressed in either release. Attaching them to `[0.2.1]` provides accurate traceability — they were known at the time of this release and carried forward intentionally.

## Regression Analysis

The three bugs fixed in this sprint share a common root cause: the v0.2.0 v3 schema sprint updated all "write path" code (serializer output, filename generation, traversal exclusion) but missed three "read path" and "rename path" consumers that locate sidecars by filename pattern or validate schema version on load.

**Why the test suite did not catch this:**
- The existing unit tests for `discover_meta2_files()` and `load_meta2()` used v2 fixtures. They correctly tested v2 behavior but never exercised v3 input.
- No integration test performed a full write-then-read cycle. The indexing tests validated output format; the rollback tests validated rollback logic from pre-built fixtures. Neither connected the two.
- The `rename_inplace_sidecar()` tests verified rename mechanics but used string-matching assertions that happened to pass because both the fixture and the code agreed on `_meta2.json`.

**Mitigation added:**
- The new `test_roundtrip.py` tests exercise the full connected pipeline. Any future schema version bump that updates output naming without updating the read/rename paths will fail these tests immediately.
