Sprint 20260401-003 completed the full v4 sidecar architecture pivot across nine phases. The implementation is functionally correct: 456 tests pass, 0 fail, and the MkDocs documentation site builds under strict mode. However, the release tag was correctly withheld because the sprint plan defined four mandatory pre-tag gates and two of them fail:

## Phase 1: Ruff Remediation

### 1.1. Purpose

Resolve all Ruff lint violations and formatting drift across the entire repository so that both `ruff check src/ tests/` and `ruff format --check src/ tests/` exit cleanly. This is the primary release blocker.

### 1.2. Spec References

- §13.2 (pyproject.toml Configuration) — Ruff configuration section defines the active rule set and per-file ignores.

### 1.3. Implementation Instructions

**1.3.1. Capture the baseline**

Before making any changes, capture the current state of lint and format violations for the after-action record:

```bash
ruff check src/ tests/ 2>&1 | tee /tmp/ruff-check-baseline.txt
ruff check src/ tests/ --statistics 2>&1 | tee /tmp/ruff-check-stats.txt
ruff format --check src/ tests/ 2>&1 | tee /tmp/ruff-format-baseline.txt
```

Record the total violation count and the per-rule breakdown from `--statistics`. This data goes into the Phase 1 session report.

**1.3.2. Apply automatic formatting**

Run the formatter first because formatting changes can resolve some lint violations (line length, whitespace) and can also shift line numbers, which would invalidate lint fix locations if done in the wrong order.

```bash
ruff format src/ tests/
```

After formatting, re-run the format check to confirm it is now clean:

```bash
ruff format --check src/ tests/
```

This MUST exit 0. If it does not, investigate and resolve any files that the formatter cannot handle.

**1.3.3. Apply automatic lint fixes**

Run Ruff's auto-fixer for all fixable violations:

```bash
ruff check src/ tests/ --fix
```

Re-run the lint check to see what remains:

```bash
ruff check src/ tests/ 2>&1 | tee /tmp/ruff-check-after-autofix.txt
```

**1.3.4. Resolve remaining lint violations manually**

Any violations that survive auto-fix require manual intervention. Work through them file by file. The active rule set is:

| Rule prefix | Category | Notes |
|---|---|---|
| E | pycodestyle errors | Line length, whitespace, syntax style |
| W | pycodestyle warnings | Deprecated syntax, trailing whitespace |
| F | pyflakes | Unused imports, undefined names, redefined variables |
| I | isort | Import ordering |
| N | pep8-naming | Naming conventions (disabled for `tests/**/*.py` via per-file-ignores) |
| UP | pyupgrade | Python 3.12+ idiom upgrades |
| B | flake8-bugbear | Common bug patterns, mutable default arguments |
| SIM | flake8-simplify | Simplifiable constructs (disabled for `tests/**/*.py` via per-file-ignores) |
| TCH | flake8-type-checking | TYPE_CHECKING import optimization (disabled for `tests/**/*.py` via per-file-ignores) |
| RUF | ruff-specific | Ruff's own lint rules |

For each remaining violation, apply one of these resolution strategies (in preference order):

1. **Fix the code.** The preferred path. If the fix is straightforward and correct, apply it.
2. **Suppress with an inline `# noqa: XXXX` comment.** Use this only when the violation is a false positive or when fixing it would change behavior in a way that requires separate testing. Always include the specific rule code, not a bare `# noqa`.
3. **Add a per-file ignore.** Use this only when an entire file legitimately needs a rule exemption (rare).

Do NOT add broad rule exclusions to `pyproject.toml` to suppress categories of violations. The existing rule set is intentional.

**Naming convention violations (N rules) in source code** deserve careful attention. If a function, variable, or parameter name violates PEP 8 naming but is part of a public API or an established internal convention, prefer an inline suppression with a brief comment explaining why. Do not rename public API surfaces in a cleanup sprint.

**1.3.5. Run the full test suite after all lint changes**

Lint fixes (especially unused-import removal, variable renames, and simplification rewrites) can introduce regressions. Run the full test suite after completing all changes:

```bash
pytest tests/ --tb=short -q
```

This MUST produce 0 failures. If any test fails, the lint fix that caused it must be reverted or corrected before proceeding.

### 1.4. Automated Verification

```bash
# Lint gate
ruff check src/ tests/
# MUST exit 0

# Format gate
ruff format --check src/ tests/
# MUST exit 0

# Regression gate
pytest tests/ --tb=short -q
# MUST produce 0 failures (skips and warnings are acceptable in this phase)
```

### 1.5. Acceptance Criteria

1. `ruff check src/ tests/` exits 0 with no output.
2. `ruff format --check src/ tests/` exits 0 with no output.
3. `pytest tests/` produces 0 test failures.
4. No broad rule exclusions added to `pyproject.toml` `[tool.ruff.lint]` configuration. Individual inline suppressions are acceptable with justification.
5. Session report records the baseline violation count, the auto-fix count, and the manual resolution count.

### 1.6. Commit Guidance

Formatting and lint fixes may be committed together or separately depending on the agent's judgment about reviewability. A reasonable approach:

```
style: apply ruff format across src/ and tests/
lint: apply ruff auto-fixes across src/ and tests/
lint: resolve remaining ruff violations manually
```

If the manual resolution set is large and spans many unrelated modules, splitting into per-module or per-rule-category commits is acceptable. The key constraint is that every commit must leave the test suite passing.

## Phase 2: Test Suite Audit

### 2.1. Purpose

Investigate the 63 skipped tests and 3 pytest warnings reported during Sprint 20260401-003 Phase 9 verification. Determine whether any skip masks a real problem, whether any warning indicates a code or configuration issue, and resolve anything that could undermine release confidence.

### 2.2. Implementation Instructions

**2.2.1. Categorize all skipped tests**

Run the test suite with skip reasons visible:

```bash
pytest tests/ -v --tb=short 2>&1 | tee /tmp/test-full-verbose.txt
```

Extract every skipped test and its skip reason. Categorize each skip into one of these buckets:

| Category | Description | Action required |
|---|---|---|
| Platform skip | Test is marked for a platform not matching the current runner (e.g., `@pytest.mark.platform_windows` on Linux) | None. These are expected. |
| ExifTool skip | Test requires ExifTool via `@pytest.mark.requires_exiftool` and ExifTool is not installed in the agent environment | None if ExifTool is genuinely unavailable. If ExifTool IS available, investigate why the mark is triggering. |
| Incomplete migration | Test was skipped during the v4 pivot because the feature it tests was being rewritten in a later phase, and the skip was never removed | Remove the skip decorator or `pytest.skip()` call. Verify the test passes. |
| Intentional deferral | Test covers functionality explicitly listed as out-of-scope or future work (e.g., `min_siblings` predicate) | Acceptable. Verify the skip reason string documents this clearly. |
| Unknown or unexplained | Test has a bare `@pytest.mark.skip` with no reason, or the reason is vague | Investigate. Either add a clear reason string, remove the skip if the test should pass, or delete the test if it is obsolete. |

Produce a categorized inventory as part of the session report. The inventory MUST account for all 63 skips.

**2.2.2. Resolve problematic skips**

For any test categorized as "incomplete migration" or "unknown/unexplained":

1. Remove the skip decorator.
2. Run the test in isolation: `pytest <test_file>::<test_name> -v --tb=long`.
3. If it passes, the skip was stale. Commit the removal.
4. If it fails, determine whether the failure is:
   - A real bug introduced during the pivot (fix it).
   - A test that references obsolete v2/v3 behavior and needs updating to v4 expectations (update it).
   - A test for functionality that was intentionally removed in v4 (delete the test).

**2.2.3. Investigate pytest warnings**

Run the test suite with warnings promoted to see their full detail:

```bash
pytest tests/ -v --tb=short -W default 2>&1 | grep -A 2 "warning" | head -60
```

For each warning, determine its source:

| Warning type | Likely cause | Action |
|---|---|---|
| `DeprecationWarning` from a dependency | Third-party library using deprecated stdlib or API | Record the dependency and version. If the warning is from a pinned dependency, note it in the report for future upgrade tracking. No code change required unless the deprecation is imminent. |
| `DeprecationWarning` from project code | Using a deprecated Python feature or stdlib API | Fix the code to use the modern equivalent. |
| `PytestUnraisableExceptionWarning` | Uncaught exception in a destructor or background thread | Investigate the source. May indicate a resource leak or teardown issue. Fix if feasible. |
| `PytestCollectionWarning` | Test discovery picking up non-test files or classes with ambiguous naming | Rename the offending file or class, or add `__test__ = False` to exclude it from collection. |
| Other | Varies | Investigate and resolve or document. |

If a warning can be resolved with a code change, resolve it. If it originates from a third-party dependency and cannot be fixed in project code, add a `filterwarnings` entry to the `[tool.pytest.ini_options]` section of `pyproject.toml` to suppress it, with a comment explaining the source and the upstream issue (if known):

```toml
filterwarnings = [
    # ExifTool wrapper uses deprecated X — tracked in pyexiftool issue #NNN
    "ignore::DeprecationWarning:pyexiftool",
]
```

**2.2.4. Validate after all changes**

```bash
pytest tests/ --tb=short -q
```

Record the updated counts: passed, skipped, warnings, failed. The session report MUST include the before and after counts and explain every delta.

### 2.3. Automated Verification

```bash
# Full suite with verbose output
pytest tests/ -v --tb=short -q

# Lint and format gates must still pass after any test file changes
ruff check src/ tests/
ruff format --check src/ tests/
```

### 2.4. Acceptance Criteria

1. Every skipped test is categorized in the session report with a clear reason.
2. No test is skipped due to incomplete v4 migration. Any such skip is either resolved (test passes) or the test is updated/removed.
3. No test has a bare `@pytest.mark.skip` without a reason string.
4. All 3 pytest warnings are investigated. Each is either resolved in code, suppressed with a documented `filterwarnings` entry, or explained in the report as a known third-party issue.
5. Zero test failures.
6. Ruff lint and format gates still pass.

### 2.5. Commit Guidance

```
tests: remove stale skip decorators from v4-migrated tests
tests: update test expectations for v4 schema behavior
tests: suppress <dependency> deprecation warning with tracking comment
tests: add skip reasons to undocumented platform-gated tests
```

Commit messages should describe what changed and why. Group related changes logically.

**3.3.4. Create the release tag**

**3.3.5. Monitor the release pipeline**

The tag push triggers `.github/workflows/release.yml`. The pipeline:

1. Checks out the tagged commit.
2. Runs the test suite on each platform (Windows x64, Linux x64, macOS x64, macOS arm64), excluding `requires_exiftool` marked tests.
3. Builds CLI and GUI executables via PyInstaller on each platform.
4. Renames artifacts with version and platform tags.
5. Creates a GitHub Release with all artifacts attached.

Monitor the GitHub Actions tab. Expected behavior:

- All four matrix jobs should pass tests and produce executables.
- The release job should create the GitHub Release with 8 artifacts (CLI + GUI for each of 4 platforms).

If the pipeline fails:

1. Read the Actions log to identify the failing step and platform.
2. If the failure is a test failure: fix the issue on `main`, delete the tag (`git tag -d v1.0.0 && git push --delete origin v1.0.0`), commit the fix, re-run gates, re-tag, and re-push.
3. If the failure is a build/packaging issue (PyInstaller, runner environment): assess whether the fix is within scope of this session. If not, document the failure in the after-action report for resolution in a follow-up.

Do NOT manually create the GitHub Release. The pipeline handles this automatically.
