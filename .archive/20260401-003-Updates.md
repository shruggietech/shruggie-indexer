## Purpose and Context

This plan implements the sidecar architecture pivot ratified on 2026-03-24 and documented in `20260401-001-sidecar-architecture-pivot.md`. The pivot fundamentally changes how `shruggie-indexer` handles sidecar files: instead of ingesting sidecar content into parent entries, the indexer now treats every file as a first-class IndexEntry and annotates believed relationships via a `relationships[]` array.

The merged technical specification (`shruggie-indexer-spec.md`, v0.1.2, 2026-04-01, AMENDED) already reflects the v4 architecture in full. This plan implements the specification.

Key deliverables across all phases:

- v4 JSON Schema file (`shruggie-indexer-v4.schema.json`) created and committed
- `relationships[]` array on IndexEntry with full annotation shape
- Sidecar rule engine with built-in rule library and TOML-based user configuration
- Removal of the sidecar ingest/reconstruct/restore pipeline
- Simplified rename (uniform for all files) and rollback (byte-perfect file copy)
- New CLI flags (`--no-sidecar-detection`, `--cleanup-legacy-sidecars`)
- Removed CLI flags (`--meta-merge`, `--meta-merge-delete`)
- GUI simplified to two operation types (Index, Rollback)
- Output suffix convention change: `_idx.json` / `_idxd.json` (permanent)
- Comprehensive test rewrite targeting v4 behavior
- Full documentation site update
- CHANGELOG update and v1.0.0 release

### Why v1.0.0

The v4 schema is a breaking change to the output contract. The project has completed all porting work, the core pipeline is stable, and the sidecar pivot represents the final architectural decision before downstream consumers (metadexer catalog) depend on the output format. Cutting v1.0.0 signals that the IndexEntry contract is stable and that semantic versioning guarantees apply in full going forward, per the versioning policy in the specification (§13.6).

class PredicateResult:
    """Result of a single predicate evaluation on a relationship rule."""
    name: str
    satisfied: bool
    pattern: str | None = None
    patterns: list[str] | None = None

    def to_dict(self) -> dict[str, Any]:

class RelationshipAnnotation:
    """A believed association between this file and another indexed file."""
    target_id: str
    type: str
    rule: str
    rule_source: str
    confidence: int
    predicates: list[PredicateResult]

    def to_dict(self) -> dict[str, Any]:
        return {
            "target_id": self.target_id,
            "type": self.type,
            "rule": self.rule,
            "rule_source": self.rule_source,

### 2.4. Implementation Instructions

**2.4.1. Rule data model**

Define dataclasses for the rule engine (in `rules.py` or a dedicated models file):

```python
@dataclass(frozen=True)
class SidecarRule:
    """A single sidecar classification rule."""
    name: str
    match: str                          # Pattern with {stem}, *, literals
    type: str                           # Relationship type (e.g., "description")
    scope: str = "file"                 # "file" or "directory"
    requires_sibling: str | None = None
    requires_sibling_any: list[str] | None = None
    excludes_sibling: str | None = None
    min_siblings: int | None = None
    enabled: bool = True
    extends: str | None = None          # Name of built-in rule to override
    rule_source: str = "builtin"        # "builtin", "user", or "pack:<name>"
```

**2.4.2. Pattern matching engine**

Implement the constrained pattern language from spec §7.5:

- `{stem}` token: matches and binds to the content file's stem (name without final extension).
- `*` token: glob-style wildcard (any sequence of non-path-separator characters).
- Literal text: exact match.

The matching function signature should be approximately:

```python
def match_rule(rule: SidecarRule, filename: str, directory_stems: set[str]) -> str | None:
    """Return the matched stem if the rule's pattern matches filename, else None."""
```

For `scope: "file"` rules, the function extracts `{stem}` from the filename and checks whether a file with that stem exists in the directory. For `scope: "directory"` rules, the `{stem}` token is not used; matching is pattern-only.

**2.4.3. Predicate evaluation**

Implement predicate evaluation per spec §7.5 and pivot §6.3:

- `requires_sibling`: check that a file matching the pattern (with `{stem}` bound) exists in the directory.
- `requires_sibling_any`: at least one of the listed patterns matches a sibling.
- `excludes_sibling`: the relationship is NOT full confidence if the pattern matches a sibling.

The `min_siblings` field is defined on the `SidecarRule` dataclass for forward compatibility but evaluation is NOT implemented in this sprint. If a rule specifies `min_siblings`, the predicate is ignored during confidence computation (treated as if it were not present). See "Items Explicitly Out of Scope" at the end of this document.

Return a `PredicateResult` list and compute the confidence code:

- 3: all predicates satisfied (or no predicates defined).
- 2: some predicates satisfied.
- 1: no predicates satisfied.

**2.4.4. Rule loading and resolution**

Implement rule loading per spec §7.5 resolution order:

1. User-defined rules (from TOML config) — evaluated first.
2. Community pack rules (from pack directory) — evaluated second.
3. Built-in rules — evaluated last.

First matching rule wins. A user rule with the same `name` as a built-in rule overrides it entirely. A rule with `enabled = false` and `extends = "builtin-name"` disables that built-in.

Implement TOML loading for user rules and packs. The built-in rules are defined as a Python data structure (list of `SidecarRule` objects).

**2.4.5. Built-in rule library**

Define the initial built-in rules matching the current sidecar type vocabulary. Derive these from the existing `metadata_identify` regex patterns in the spec (§7.3) and the pivot document §6.1 examples. At minimum:

- `yt-dlp-description`: `{stem}.description` → type `description`
- `yt-dlp-info`: `{stem}.info.json` → type `json_metadata`
- `yt-dlp-thumbnail`: `{stem}_screen.jpg` → type `screenshot`, with `requires_sibling` predicate
- `yt-dlp-subtitles`: `{stem}.*.vtt` → type `subtitles`, with `requires_sibling_any` predicate
- `any-lnk`: `*.lnk` → type `shortcut`, scope `directory`
- `desktop-ini`: `desktop.ini` → type `desktop_ini`, scope `directory`
- `any-url`: `*.url` → type `link`
- `any-torrent`: `*.torrent` → type `torrent`
- `nfo-file`: `{stem}.nfo` → type `generic_metadata`
- `hash-file`: `{stem}.md5`, `{stem}.sha1`, `{stem}.sha256` (multiple rules) → type `hash`

Consult the existing `metadata_identify` patterns in the codebase for the full set. Every pattern currently handled by the regex system should have a corresponding built-in rule.

**2.4.6. Classifier entry point**

Create the top-level classification function:

```python
def classify_relationships(
    entries: list[IndexEntry],
    rules: list[SidecarRule],
) -> dict[str, list[RelationshipAnnotation]]:
    """Classify relationships for all entries.

    Returns a mapping of entry ID to list of RelationshipAnnotation objects.
    Entries with no matches are absent from the result (not present with empty list).
    """
```

This function:

1. Groups entries by parent directory.
2. For each directory, collects the set of filenames and stems.
3. For each file in the directory, iterates rules in resolution order.
4. On first match, evaluates predicates, computes confidence, records predicate detail.
5. Resolves `target_id` by finding the entry whose stem matches the bound `{stem}` value.
6. Returns the annotation map.

### 3.4. Implementation Instructions

**3.4.1. Remove sidecar exclusion from discovery**

In the file discovery/traversal module (`indexer.py` or equivalent):

- Remove the logic that identifies files as sidecars and excludes them from entry building. Search for log messages like `"Excluded N sidecar file(s) from entry building"` and the code that produces them.
- All files that pass `metadata_exclude_patterns` filtering now enter the main entry list. The only files excluded are the indexer's own output artifacts (matched by `metadata_exclude_patterns`).
- Remove the `metadata_identify` regex compilation and matching. This regex set was used to identify sidecar types during discovery. It is replaced by the rule engine's post-discovery classification.

**3.4.2. Add relationship classification pass**

After all IndexEntry objects are constructed (after the existing entry-building loop), add a new pipeline stage:

```python
if not config.no_sidecar_detection:
    rules = load_rules(config)  # user + packs + built-in
    relationship_map = classify_relationships(entries, rules)
    for entry in entries:
        if entry.id in relationship_map:
            entry.relationships = relationship_map[entry.id]
```

Note: `IndexEntry` is NOT a frozen dataclass (unlike `IndexerConfig`), so direct field assignment is valid here. The mutation occurs before serialization, consistent with the incremental construction pattern described in spec §9.4.

**3.4.3. Respect no_sidecar_detection flag**

When `config.no_sidecar_detection` is `True`, skip the entire classification pass. No rules are loaded, no relationships are computed, no `relationships` field appears on any entry.

**3.4.4. Update entry construction steps**

Per spec §6.8:

- Remove Step 9 (sidecar content ingestion into metadata array).
- Simplify Step 10 (metadata array construction): the metadata array now contains only ExifTool entries (when `--meta` is enabled). No sidecar-origin MetadataEntry objects.
- Add Step 14: relationship annotation (performed as a batch pass after all entries are constructed, not per-entry).

**5.4.1. Simplify rollback engine**

In `rollback.py`:

- Remove `_plan_sidecar_restore()` or equivalent sidecar-specific restore planning.
- Remove `_decode_sidecar_data()` — no sidecar content decoding for v4 entries.
- Remove `_restore_json()` — no JSON re-serialization.

- Remove `sidecar_data`, `sidecar_binary`, and `metadata_entry` fields from `RollbackAction` (or equivalent action dataclass).

Simplified v4 rollback behavior: every entry in the rollback plan is a file copy. The source file exists on disk in its storage-named form. The rollback engine copies it to the original relative path (`file_system.relative`) and restores timestamps. Uniform for all files regardless of former sidecar status.

**5.4.2. Retain legacy v2/v3 backward compatibility**

- The legacy path is invoked ONLY when loading `_meta2.json` or `_meta3.json` input files with `schema_version` < 4.
- New `_idx.json` output NEVER triggers the legacy path.

**5.4.3. Implement legacy output cleanup**

Create `cleanup.py` (or add to an existing utility module):

The `cleanup_legacy_sidecars` feature, when enabled:

1. After a successful index run that writes `_idx.json` and/or `_idxd.json` files, scan the same directories for old-format siblings (`_meta.json`, `_meta2.json`, `_meta3.json`, `_directorymeta.json`, `_directorymeta2.json`, `_directorymeta3.json`).
2. Delete an old-format file ONLY if a new-format file was successfully written in the same run for the same scope.
3. NEVER delete orphaned legacy files in directories the current run did not touch.
4. Log each deletion at `INFO` level.

Wire this into the indexer pipeline: after successful in-place output, if `config.cleanup_legacy_sidecars` is `True`, invoke the cleanup.

```markdown
## [1.0.0] - 2026-XX-XX

### Changed

- **BREAKING: v4 schema.** Output schema version bumped from 2 to 4.
  Every file is now a first-class IndexEntry with its own identity hash,
  timestamps, and metadata. Sidecar files are no longer embedded into
  parent entries.
- **BREAKING: Output suffix convention.** In-place output files now use
  `_idx.json` (file-level) and `_idxd.json` (directory-level) as permanent
  suffixes, replacing the version-bumped `_meta2.json` / `_directorymeta2.json`
  convention.
- **BREAKING: Removed CLI flags.** `--meta-merge` and `--meta-merge-delete`
  have been removed. These operations are no longer part of the indexer's
  responsibility.
- **BREAKING: Removed GUI operations.** Meta Merge and Meta Merge Delete
  operation types removed from the desktop application. The operation
  dropdown now contains Index and Rollback.

### Added

- **Relationship annotations.** The indexer now annotates believed
  associations between files via a `relationships[]` array on the
  sidecar-like file's IndexEntry. Annotations include target ID,
  relationship type, rule name, rule source, confidence level, and
  predicate evaluation detail.
- **Sidecar rule engine.** A TOML-based rule engine for classifying
  sidecar relationships. Ships with a built-in rule library covering
  common conventions (yt-dlp, .lnk, .url, desktop.ini, .torrent,
  .nfo, hash files). Users can define custom rules, override built-in
  rules, or disable rules via configuration.
- **`--no-sidecar-detection` CLI flag.** Disables relationship
  classification entirely. Produces a pure filesystem inventory.
- **`--cleanup-legacy-sidecars` CLI flag.** Removes obsolete tool output
  files when new-format replacements are written.
- **Community rule pack support.** TOML-based rule packs can be installed
  to the user's pack directory for shared rule libraries.
- **Simplified rollback.** All files roll back as uniform file copies.
  Byte-perfect restoration by construction. Legacy v2/v3 rollback
  supported via backward-compatible fallback path.

### Removed

- Sidecar content ingest/reconstruct/restore pipeline.
- `_decode_sidecar_data()`, `_restore_json()`, `_apply_text_encoding()`
  sidecar reconstruction codepaths.
- Sidecar-rename tracking (all files rename uniformly).
- MetaMergeDelete execution and safety logic.
- Sidecar-origin fields on MetadataEntry (`format`, `transforms`,
  `json_style`, `json_indent`, sidecar-specific `encoding`,
  `file_system`, `size`, `timestamps`).
- `metadata_identify` regex pattern system (replaced by rule engine).
```

```bash
# 1. Verify all tests pass

# 7. Push commit and tag
git push && git push --tags
```

The tag push triggers the GitHub Actions release pipeline (`.github/workflows/release.yml`), which:

1. Checks out the tagged commit.
2. Runs the test suite on each platform.
3. Builds CLI and GUI executables via PyInstaller for Windows (x64), Linux (x64), macOS (x64), and macOS (arm64).
4. Renames artifacts with version and platform tags.
5. Creates a GitHub Release with the artifacts attached.

**Do NOT manually create the GitHub Release.** The pipeline handles this automatically. Monitor the Actions tab to confirm all jobs succeed.

If the pipeline fails:

- Check the Actions log for the failing step.
- Fix the issue on `main`.
- Delete the tag (`git tag -d v1.0.0 && git push --delete origin v1.0.0`).
- Re-run the version bump and tag procedure.
